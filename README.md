> **MVP** — built to demonstrate LLM-first data exploration, semantic clustering,
> taxonomy mapping, and LangChain retrieval/chunking modules on unstructured
> financial text. Each component maps directly to a skill in the target job description.

> **Data notice:** All credit memos in this repository are **fully synthetic** —
> generated by GPT-4o using realistic leveraged loan structures, issuer names,
> and financial metrics. No real issuer data, proprietary research, or client
> information is used anywhere in the pipeline.

---

## What it does

Ingests synthetic credit memos for leveraged loan issuers and turns them into a
searchable intelligence layer. Given a free-text query, the system returns
semantically similar historical memos, their auto-discovered taxonomy labels,
and the structured analytical evidence that justifies each label.

**The key capability — combined structured + semantic search:**

```python
# "Find memos like this one, but only where leverage > 7x and collateral = first lien"
retriever.query(
    text="covenant headroom deteriorating, sponsor unlikely to support",
    filters={"net_leverage": {"$gt": 7.0}, "collateral": "first_lien"},
)
```

This is not possible with keyword search alone. Semantic similarity finds
conceptually related memos; metadata filters restrict to structurally relevant ones.

## Method explanation

### Why LLM-first before embeddings?

Imposing a taxonomy top-down means missing emergent patterns in the data.
GPT-4o reads a stratified 200-doc sample at `temperature=0` and proposes
structure from the data itself — risk themes, actions, and outcomes — including
flagging cases where no seed label fits and proposing new ones.

This prototype costs ~$4 total. Labeling all 3,000 docs with GPT-4o would cost ~$60.
BGE embeddings then scale the discovered structure to the full corpus at zero
marginal cost per document.

### Why section-aware chunking?

A credit memo is not a flat document. Naive token-count chunking destroys
section boundaries — a chunk containing half a cap table and half a risk
narrative answers no query well. Section-aware chunking preserves semantic
coherence: each chunk answers a specific analytical question.

Financial tables (cap table, summary of terms) are serialized to analyst prose
by GPT-4o before embedding — embedding models are trained on language, not
tabular layout.

### Why combined structured + semantic search?

Semantic similarity alone finds conceptually related memos but can't enforce
structural constraints. A query about "covenant stress" may return a mix of
leveraged and investment-grade issuers. Adding `{"net_leverage": {"$gt": 7.0}}`
restricts results to the structurally relevant subset while preserving semantic
ranking. ChromaDB stores memo-level metrics on every chunk so any chunk can be
filtered by any memo attribute without a join.

### Cluster purity as validation

After KMeans clustering, we measure how well clusters align with GPT-4o labels
using adjusted Rand index (ARI). High ARI means the embedding space captures
the same semantic structure as the LLM — validating that BGE embeddings are
appropriate for this domain before scaling.

## Pipeline

```
Phase 1  generate_corpus.py    Synthetic corpus via OpenAI GPT-4o
Phase 2  normalize.py          Section-aware chunking (LangChain module)
Phase 3  llm_explorer.py       Taxonomy discovery + functional decomposition
Phase 4  embed_cluster.py      BGE embeddings + KMeans + UMAP + ChromaDB
Phase 5  retriever.py          LangChain VectorStoreRetriever
Phase 6  app/app.py            Streamlit demo UI
```

## Setup

### Prerequisites

- Python 3.11+
- OpenAI API key **or** Azure OpenAI deployment

### Install

```bash
git clone https://github.com/GenAIFinance/credit-memo-intelligence.git
cd credit-memo-intelligence
pip install -r requirements.txt
```

### Configure credentials

```bash
cp .env.example .env
```

Edit `.env` — choose one provider:

```bash
# Option A — OpenAI personal API
OPENAI_API_KEY=sk-proj-...

# Option B — Azure OpenAI
AZURE_OPENAI_ENDPOINT=https://<resource>.openai.azure.com/
AZURE_OPENAI_API_KEY=<key>
AZURE_OPENAI_DEPLOYMENT=<deployment-name>
```

Set the active provider in `config/config.yaml`:

```yaml
llm:
  provider: "openai"   # or "azure"
```

### Run pipeline

```bash
# Quick demo run (100 docs, ~$0.15 in API cost)
python -m src.generation.generate_corpus --target 100
python -m src.preprocessing.normalize
python -m src.exploration.llm_explorer --sample 20
python -m src.embeddings.embed_cluster
python -m src.retrieval.retriever --build
python -m streamlit run app/app.py

# Full run (3,000 docs, ~$4 in API cost)
python -m src.generation.generate_corpus
python -m src.preprocessing.normalize
python -m src.exploration.llm_explorer
python -m src.embeddings.embed_cluster
python -m streamlit run app/app.py
```
